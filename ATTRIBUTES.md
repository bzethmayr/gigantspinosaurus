Attribute DSL Description (for gpt‑oss:20b)

1. Core Idea

   This DSL defines attribute constructors for
   building a canonical attribute definition mapping: `Map<String, Function<T, byte[]>>`
   initially populated by `Consumer<Map<String, Function<T, byte[]>>>` implementations.

   Each entry associates an attribute name with a function that,
   given an instance of `T`, produces the canonical byte representation of that attribute.
    
   The DSL is intentionally functional, compositional, and context‑free:
   no mutation leaks, no side effects, no implicit global state.
   An example consumer is `BoundAttributes<T>` - it uses the DSL to populate attribute bindings for a class.
   After population, the map is sealed and never exposed again.


2. `AttributeBinder<T>`

   `AttributeBinder<T>` is an optional shorthand for the “builder” abstraction. 
   It is not a standalone type with its own methods;
   instead it is a duck‑typed specialization of: `Consumer<Map<String, Function<T, byte[]>>>`
   An `AttributeBinder<T>`-compatible method's only job
   is to populate the map with one or more `(String → Function<T, byte[]>)` entries.
   
   Meaning, a binder is simply:
   
   * a consumer that accepts a mutable map
   * populates it with attribute definitions
   * and returns nothing

   Why this matters? Because the rest of the API deals in `Consumer<Map<String, Function<T, byte[]>>>`,
   the DSL can treat binders as first‑class combinators:
   
   they can be
   * composed
   * wrapped
   * generated by higher‑order functions
   * add derived attributes                     
   * add attributes from other binders

   This is the foundation for the DSL’s “pipeline” feel.


3. `binder(...)`

   binder is a duck‑type acknowledgement operator.
   
   Signature (conceptual): `binder(Consumer<Map<String, Function<T, byte[]>>>) → AttributeBinder<T>`
   
   Semantics: `binder` does not mutate or finalize anything. It simply returns the same method reference it was given.
   
   `binder` exists to:
   * make it easier to remember the generic type
   * mark a boundary in the DSL
   * allow composition
   * allow the caller to treat the returned value as a binder without caring about its internal representation
   
   Think of it as a semantic wrapper, not a transformation.


4. `adds(...)` and its family of overloads
   
   `adds` is one of several overloads that insert data into containers.
   
   The overload relevant to your example:
   ```java
   public static <K, V> Consumer<Map<K, V>> adds(final K key, final V value) {
           return m -> m.put(key, value);
       }
   ```

   ```text
   adds(String attributeName, Function<T, byte[]> encoder)
   → Consumer<Map<String, Function<T, byte[]>>>
   ```
   Semantics:
   
   The returned binder, when applied to the map, performs:
   ```
   map.put(attributeName, encoder);
   ```
   No more, no less.
   
   Family behavior: Other `adds` overloads may:
   * deal with arbitrary collections instead of maps
   * deal with arbitrary consumers or bi-consumers
   * ...
   Since we deal with maps, we can use the generic map overload, `adds(K,V)`.

5. Example: Version Field Construction
   
   Given the line:
   
   `return binder(adds(VERSION_FIELD, fromShort(T::version)));`
   
   A gpt‑oss:20b‑level model should understand the following:
   
   Step‑by‑step meaning
   
   1. `T::version` is a method reference (e.g., a `short` getter).
   2. `fromShort(...)` wraps this into a canonical byte encoder:
   3. `Function<T, byte[]>` encodes the method result for any given `T`
   4. `adds(VERSION_FIELD, encoder)` produces a consumer that inserts:
   5. VERSION_FIELD → encoder
   6. `binder(...)` wraps that binder and returns it unchanged.
   
   Net effect: The entire expression returns an AttributeBinder<T> that,
   when invoked, adds a single entry to the attribute map:
   
   `VERSION_FIELD → (ctx -> canonicalShortBytes(T::version))`

 
6. DSL Philosophy
   
   The DSL is built around three principles:
   
   1. Compositionality
      
      Every operator returns an `Consumer<Map<String, Function<T, byte[]>>>` or a function that produces one.
      This allows varargs uses like:
      ```
      clientConstructor(
      binder(adds("foo", encodeFoo)),
      adds("bar", encodeBar),
      addsVersion()
      )
      ```
   2. Canonicalization
      
      All encoders (`fromShort`, `fromInt`, `fromBytes`, etc.) produce canonical byte sequences given an instance.
      This ensures deterministic serialization and stable attestation.
      
   3. No implicit context
      
      All attribute functions are:
      
      `Function<T, byte[]>`
      
      They receive the instance explicitly. No global state, no hidden dependencies.


7. Grammar
   ```
   AttributeBinder<T> ::= Map<String, Function<T, byte[]>> -> void
   binder(binder) ::= binder
   adds(name, encoder) ::= AttributeBinder<T>
   encoder ::= Function<T, byte[]>
   fromShort(shortConst) ::= encoder
   fromInt(intConst)     ::= encoder 
   fromBytes(bytes)      ::= encoder
   ```


8. Summary
   
   A model like gpt‑oss:20b should walk away with:
   * binder = identity wrapper for binders
   * adds = map‑population combinator
   * AttributeBinder<T> = a Consumer of a map
   * encoders = pure functions producing canonical bytes
   * the entire DSL = a functional pipeline for constructing attribute maps
